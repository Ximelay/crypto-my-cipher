# XiCipher

## Что это

XiCipher — мой потоковый шифр, написанный в рамках задания. Суть простая: берем пароль, из него получаем ключ, генерируем поток псевдослучайных байт и XOR-им с текстом. Плюс поверх этого идет подстановка через S-блок для дополнительной запутанности.

Шифр симметричный — один и тот же пароль используется и для шифрования, и для расшифровки.

## Параметры

- Ключ: 256 бит (SHA-256 от пароля)
- Nonce: 8 байт (генерируется случайно при каждом шифровании)
- Seed для ГПСЧ: 32 байта (SHA-256 от ключа + nonce)
- Блок ГПСЧ: 32 байта (8 слов по 4 байта)
- Раундов в ГПСЧ: 8
- S-блок: 256 байт

## Как работает шифрование

Весь процесс по шагам:

1. Из пароля через SHA-256 получаем `key_bytes` (32 байта)
2. Генерируем случайный `nonce` (8 байт) — он нужен чтобы один и тот же текст с одним паролем давал разный результат
3. Считаем `seed = SHA-256(key_bytes + nonce)`
4. Скармливаем seed в наш ГПСЧ, он выдает keystream нужной длины
5. Из key_bytes строим S-блок через RC4-KSA
6. Для каждого байта текста: XOR с байтом keystream, потом прогоняем через S-блок

Формула:
```
C[i] = SBox[ P[i] XOR K[i] ]
```

## Расшифровка

Все то же самое, только в обратную сторону:
```
P[i] = SBox_inv[ C[i] ] XOR K[i]
```

Сначала обратная подстановка через инвертированный S-блок, потом XOR. Работает потому что keystream воспроизводится из того же пароля и nonce, а S-блок — это перестановка, значит он обратим.

## Разбор компонентов

### Получение ключа

Тут все просто — хешируем пароль через SHA-256 и получаем 32 байта. Какой бы длины ни был пароль, ключ всегда 32 байта. Плюс SHA-256 дает лавинный эффект — поменяй один символ в пароле, и ключ полностью другой.

### Nonce

8 случайных байт, генерируются при каждом шифровании. Передаются открыто вместе с шифротекстом. Без nonce одинаковый текст + одинаковый пароль давали бы одинаковый шифротекст, что плохо.

### ГПСЧ (xi_prng)

Вдохновлен ChaCha20, но упрощен. Работает так:

Берем seed (32 байта), разбиваем на 8 чисел по 4 байта. Это начальное состояние.

Для каждого блока выхода:
- Копируем состояние
- XOR-им последнее слово со счетчиком блока (чтобы каждый блок был уникальный)
- Прогоняем 8 раундов смешивания
- Складываем результат с начальным состоянием

Каждый раунд — это ARX (Add, Rotate, XOR) над парами слов:
```
a = (a + b) mod 2^32
b = RotateLeft(b, shift) XOR a
```

Сдвиги для четырех пар: 7, 9, 13, 18 бит. После парного смешивания идет диагональное:
```
s0 ^= s4;  s2 ^= s6
s1 ^= s5;  s3 ^= s7
```

Зачем ARX: эти три операции (сложение, сдвиг, XOR) хорошо перемешивают биты и быстро считаются. А финальное сложение с начальным состоянием делает преобразование необратимым — нельзя по выходу восстановить промежуточное состояние.

### S-блок

Строится через RC4-KSA (Key Scheduling Algorithm). По сути перемешиваем массив [0, 1, 2, ..., 255] в зависимости от байтов ключа:

```
s = [0, 1, 2, ..., 255]
j = 0
for i in range(256):
    j = (j + s[i] + key[i % len(key)]) % 256
    swap(s[i], s[j])
```

Получается таблица подстановки на 256 значений. Каждый вход дает ровно один выход (биекция), поэтому можно построить обратную таблицу для расшифровки.

Зачем S-блок нужен: он добавляет нелинейность. XOR сам по себе линейная операция, и если злоумышленник как-то узнает часть keystream, он может по XOR восстановить текст. А S-блок ломает эту линейность.

### Что на выходе

После шифрования получаем:
- `ciphertext` — шифротекст в Base64
- `nonce` — hex-строка, 8 байт
- `algo` — строка "XiCipher-v1"

Для расшифровки нужны все три поля + пароль.

## Сравнение с другими шифрами

| | XiCipher | ChaCha20 | RC4 | AES-CTR |
|---|---|---|---|---|
| Тип | Потоковый | Потоковый | Потоковый | Блочный (CTR) |
| ГПСЧ | ARX, 8 раундов | ARX, 20 раундов | нет | SPN |
| S-блок | Из ключа (RC4-KSA) | Нет | Нет | Фиксированный |
| Ключ | 256 бит | 256 бит | 40-2048 бит | 128/192/256 бит |
| Nonce | 64 бита | 96 бит | Нет | 128 бит |

По сравнению с ChaCha20 у нас меньше раундов (8 против 20), но есть дополнительный S-блок. В отличие от AES, S-блок зависит от ключа, а не зашит в алгоритм. RC4 вообще не использует nonce, что считается его слабостью.

## API

Все завернуто в REST API на FastAPI:

- `GET /` — главная страница с описанием
- `GET /info` — параметры алгоритма в JSON
- `POST /keygen` — генерация случайного пароля
- `POST /encrypt` — шифрование (принимает text и password, возвращает ciphertext + nonce)
- `POST /decrypt` — расшифровка (принимает ciphertext, nonce, password)
- `GET /docs` — Swagger, там можно все потыкать руками
